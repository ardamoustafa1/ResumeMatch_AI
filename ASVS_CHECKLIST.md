# OWASP ASVS v4.0.3 Checklist - NetworkForge

This checklist is based on the OWASP Application Security Verification Standard (ASVS) Level 1 & Level 2 criteria.

## V2: Authentication Verification Requirements
- [x] 2.1.1: Verify user passwords are encrypted/hashed using modern algorithms (Argon2id/Bcrypt).
- [x] 2.1.7: Verify passwords are submitted via secure POST requests, not query parameters.
- [ ] 2.3.1: Verify session tokens are rotated after login and logout. (Refresh Token Rotation pending)
- [x] 2.7.1: Verify application uses secure session IDs generated by a CSPRNG.
- [x] 2.8.2: Verify API keys are hashed in the database, and only shown once upon creation.

## V3: Session Management Verification Requirements
- [x] 3.1.1: Verify application never reveals session tokens in URL parameters. (WebSocket uses short-lived ticket)
- [x] 3.4.1: Verify cookie-based session tokens use `Secure`, `HttpOnly`, and `SameSite` attributes.
- [ ] 3.7.1: Verify application ensures concurrent session limits and anomaly detection.

## V4: Access Control Verification Requirements
- [x] 4.1.1: Verify application enforces least privilege principles for all users.
- [x] 4.3.1: Verify users can only access data belonging to them. (Tested via endpoint tests)
- [ ] 4.3.2: Verify data export/deletion functionality securely handles cascading records.

## V5: Validation, Sanitization and Encoding Verification Requirements
- [x] 5.1.1: Verify all input relies on positive validation (allow-listing) via Pydantic.
- [x] 5.1.4: Verify input bounds are enforced (Length limits on CV/JD texts).
- [ ] 5.3.4: Verify uploaded files are validated for content type and size limits.

## V7: Error Handling and Logging Verification Requirements
- [x] 7.1.1: Verify no sensitive info (PII/Tokens) is included in application logs.
- [x] 7.4.1: Verify application uses centralized error logging (Sentry).
- [ ] 7.4.2: Verify Sentry has a dedicated scrubber to prevent PII leakage.

## V8: Data Protection Verification Requirements
- [x] 8.1.1: Verify sensitive data is transmitted via TLS 1.2/1.3.
- [x] 8.2.1: Verify passwords and secrets are not stored in source code.
- [ ] 8.3.1: Verify user data retention complies with GDPR. (Purge task exists, needs integration test)

## V10: Malicious Code Verification Requirements
- [x] 10.1.1: Verify codebase is scanned for known vulnerabilities (Dependabot, npm audit).
- [ ] 10.3.1: Verify source code is analyzed via SAST (Bandit, CodeQL).
- [ ] 10.3.2: Verify container images are scanned for vulnerabilities (Trivy).
